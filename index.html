Теория: JCommander — параметры командной строки

▼

В твоём проекте уже используется JCommander — библиотека для разбора аргументов командной строки. Сейчас в AppArgs уже есть --tasks-file, --categories-file, --users-file, --port.

Нужно добавить ещё один параметр — --secret. Это будет секретный ключ, которым подписываются JWT-токены. Его нельзя хранить в исходном коде — только передавать снаружи.

Аналогия

Представь, что ты директор завода. Каждому сотруднику выдаётся пропуск (токен). Пропуска подписывает только директор своей личной печатью (secret). Охранник (сервер) проверяет печать и пропускает тебя. Секрет — это твоя печать: если она утечёт, любой сможет подделать пропуска.

В JCommander аннотация @Parameter принимает несколько опций:

  • names — список допустимых имён флага, например ["--port", "-p"]
  • required = true — если параметр не передан, JCommander бросит исключение
  • required = false (по умолчанию) — параметр опциональный, будет использоваться defaultValue

▼

Enum (перечисление) — тип данных с фиксированным набором значений. Идеально подходит для ролей, т.к. роль — это всегда одно из предопределённых значений.

Пример: роли в другом приложении Kotlin 
// Пример с сервисом доставки еды (не твоё приложение)
enum class DeliveryRole {
    Customer,       // обычный покупатель
    Courier,        // курьер
    RestaurantAdmin // администратор ресторана

// Использование в data class:

// Работа с enum: получение по строке из CSV
val role = DeliveryRole.valueOf("Courier")  // DeliveryRole.Courier
val roleName = role.name  // строка "Courier"
 
// Когда используем when — Kotlin требует покрыть все варианты:

Создание и верификация JWT-токенов

🔐

Теория: что такое JWT и зачем он нужен

▼

JWT (JSON Web Token) — это способ «упаковать» данные о пользователе в строку и подписать эту строку, чтобы никто не мог её подделать.

Аналогия — выдача пропуска на мероприятие

Охрана (сервер) выдаёт пропуск (токен) с именем гостя (userId), датой истечения и печатью (подписью). Чтобы войти, показываешь пропуск — охрана смотрит на печать и дату. Не нужно снова проверять паспорт (не нужна база данных сессий).

Токен выглядит так — три части разделены точками:

Пример JWT-токенаJWT
eyJhbGciOiJIUzUxMiIsInR5cCI6IkpXVCJ9.eyJzdWIiOiJkZTQyYTAwZiIsImlzcyI6ImFwcCIsImV4cCI6MTczNTY4OTYwMH0.SflKxwRJSMeKKF2QT4fwpMeJf36POk6yJV_adQssw5c
│──── HEADER (алгоритм) ────│──────────── PAYLOAD (данные) ─────────────│─── SIGNATURE (подпись) ───│

Payload расшифровывается в:

Содержимое payloadJSON
{
  "sub": "de42a00f-7f43-4d10-808d-bee47fdeef49",  // ID пользователя
  "iss": "tasks-app",                              // кто выдал
  "exp": 1735689600                                 // до каких пор действителен
}
🚫
Payload читается без ключа! Base64 — это не шифрование. Никогда не кладите в payload пароли, секреты или конфиденциальные данные.

Шаг 2 (пример не из твоей работы): Представим приложение-заметки "NoteApp". Там тоже нужен JWT. Класс будет выглядеть так:

NoteJwtTools.kt (пример) Kotlin 
import com.auth0.jwt.JWT
import com.auth0.jwt.JWTVerifier
import com.auth0.jwt.algorithms.Algorithm
import com.auth0.jwt.exceptions.JWTVerificationException
import java.time.Instant
 
class NoteJwtTools(private val secret: String) {
 
    // Algorithm и Verifier создаём ОДИН РАЗ для всего приложения
    private val algorithm = Algorithm.HMAC512(secret)
 
    private val verifier: JWTVerifier = JWT.require(algorithm)
        .withIssuer("note-app")
        .build()
 
    // Создать токен для пользователя. Builder создаём КАЖДЫЙ РАЗ.
    fun createToken(userId: String): String =
        JWT.create()
            .withIssuer("note-app")
            .withSubject(userId)              // кто владелец токена
            .withExpiresAt(Instant.now().plusSeconds(3600)) // живёт 1 час
            .sign(algorithm)
 
    // Проверить токен. Вернуть userId или null если невалидный.
    fun getUserId(token: String): String? =
        try {
            verifier.verify(token).subject  // вернёт "sub" из payload
        } catch (e: JWTVerificationException) {
            null  // неверная подпись, истёк, или другая ошибка
        }
}
✅
Ключевое правило: Algorithm и JWTVerifier — создавай один раз в конструкторе класса. JWT.create() — вызывай каждый раз при создании нового токена.

Как это использовать в том же NoteApp (в main функции):

Пример использования JwtToolsKotlin
val jwt = NoteJwtTools(appArgs.secret)
 
// Создать токен — например при логине:
val token = jwt.createToken("de42a00f-7f43-4d10-808d-bee47fdeef49")
// → "eyJhbGciOiJIUzUxMiIsInR5cCI6IkpXVCJ9..."
 
// Проверить токен — например при каждом запросе:
val userId = jwt.getUserId(token)
// → "de42a00f-7f43-4d10-808d-bee47fdeef49" или null

📝 Что делаешь в своём проекте на этом этапе

  • В build.gradle.kts добавь зависимость com.auth0:java-jwt:4.4.0 в блок dependencies
  • Создай файл src/main/kotlin/ru/yarsu/auth/JwtTools.kt — класс принимает secret: String, имеет методы createToken(userId: String): String и getUserId(token: String): String?
  • В withIssuer() поставь любое осмысленное имя, например "tasks-app" — главное использовать одно и то же везде
  • Убедись что createToken выбрасывает исключение при ошибке, а getUserId возвращает null при любой ошибке валидации

http4k Filters + RequestKey

Фильтр аутентификации: извлечение токена из запроса

🔗

Теория: как работают фильтры в http4k

▼

В твоём проекте уже есть фильтр — lensFailureFilter. Фильтр в http4k — это функция, которая оборачивает обработчик: перехватывает запрос до обработчика и/или ответ после.

Аналогия — охранник на входе

Фильтр — это охранник перед кабинетом. Директора (обработчик) нельзя беспокоить без проверки. Охранник смотрит пропуск (токен), записывает имя посетителя в журнал (RequestKey), и только потом пускает к директору или разворачивает обратно (401).

Анатомия Filter в http4kKotlin
val myFilter = Filter { next ->   // next — это следующий обработчик
    { request ->               // это лямбда, которая вызывается на каждый запрос
 
        // Код ДО обработчика — здесь читаем/проверяем запрос
        val result = next(request)  // передаём запрос дальше
        // Код ПОСЛЕ обработчика — здесь можно изменить ответ
 
        result  // возвращаем ответ
    }
}

Чтобы ранний возврат (не пустить дальше), просто не вызываем next():

Фильтр с ранним возвратом (пример)Kotlin
val checkTimeFilter = Filter { next ->
    { request ->
        val hour = LocalTime.now().hour
        if (hour < 9 || hour > 18) {
            // Сервис закрыт — next() НЕ вызываем!
            Response(Status.SERVICE_UNAVAILABLE)
                .body("Сервис работает только с 9 до 18")
        } else {
            next(request)  // В рабочее время — пускаем дальше
        }
    }
}

Теория: RequestKey — передача данных между фильтром и обработчиком

▼

Фильтр извлёк userId из токена. Как передать его в обработчик? Нельзя использовать глобальные переменные (параллельные запросы сломают всё). Решение — RequestKey: типобезопасный «карман» внутри объекта Request.

Поток обработки запроса
HTTP Request
→
Filter
читает токен,
кладёт userId
→
Request
+ userId
→
Handler
достаёт userId
из Request
RequestKeys.kt (пример для NoteApp)Kotlin
import org.http4k.lens.RequestKey
 
// RequestKey.optional — userId может отсутствовать (для публичных маршрутов)
val userIdKey = RequestKey.optional<String>("userId")
 
// Как положить значение в Request:
val enriched = request.with(userIdKey of "de42a00f-...")
 
// Как достать значение в обработчике:
val userId: String? = userIdKey(request)  // null если не было токена
ℹ️
optional vs required: используй RequestKey.optional для userId — токена может не быть у публичных маршрутов. Если нужно гарантировать наличие, используй required — но тогда запрос без этого ключа бросит исключение.
🛡️

Полный пример фильтра аутентификации (не из твоей работы)

▼

NoteApp — сервис заметок. Клиент передаёт токен в заголовке Authorization: Bearier <token>. Фильтр извлекает userId и кладёт в контекст запроса:

NoteAuthFilter.kt (пример)Kotlin
import org.http4k.core.Filter
import org.http4k.core.Response
import org.http4k.core.Status
import org.http4k.core.with
 
fun makeAuthFilter(
    jwt: NoteJwtTools,
    userIdKey: /* RequestKey */
): Filter = Filter { next ->
    { request ->
        // 1. Читаем заголовок Authorization
        val authHeader: String? = request.header("Authorization")
 
        // 2. Проверяем что начинается с "Bearier " и убираем префикс
        val token: String? = authHeader
            ?.takeIf { it.startsWith("Bearier ") }
            ?.removePrefix("Bearier ")
 
        // 3. Проверяем токен — получаем userId или null
        val userId: String? = token?.let { jwt.getUserId(it) }
 
        // 4. Кладём userId в Request (даже если null — для публичных маршрутов)
        val enriched = request.with(userIdKey of userId)
 
        // 5. Передаём обогащённый запрос дальше
        next(enriched)
    }
}
✅
Этот фильтр не возвращает 401 — он только кладёт userId (или null) в контекст. Возврат 401 происходит позже — в фильтре проверки прав или в самом обработчике, где нужна аутентификация.

📝 Что делаешь в своём проекте на этом этапе

  • Создай файл src/main/kotlin/ru/yarsu/auth/RequestKeys.kt — определи val userIdKey = RequestKey.optional<String>("userId")
  • Создай файл src/main/kotlin/ru/yarsu/auth/AuthFilter.kt — функция makeAuthFilter(jwt: JwtTools, userIdKey: ...): Filter
  • В фильтре: читай заголовок "Authorization", убирай префикс "Bearier " (именно с такой опечаткой, как в спецификации!), верифицируй через jwt.getUserId(), кладь в Request через request.with(userIdKey of userId)
  • Фильтр пока не возвращает 401 — только обогащает запрос. 401 будет в следующем этапе.

RBAC — ролевая модель

Система прав (Permissions) и фильтр авторизации

🎯

Теория: зачем отделять роль от прав?

▼

Роль пользователя (UserRole) — это категория из базы данных. Права (Permissions) — это конкретный список что можно делать в приложении. Разделение важно потому что:

  • Один класс Permissions описывает бизнес-логику прав — независимо от того, откуда пришли данные
  • Легко добавить новую роль или изменить её права — только в одном месте
  • Обработчики работают с Permissions, а не с конкретными ролями — меньше связность
Аналогия — карта доступа в офисе

В базе хранится твоя должность (роль): «Стажёр». При входе карточку прикладываешь — турникет знает что Стажёр может ходить в переговорку и столовую, но не в серверную. Права — это список дверей, а не сама должность.

Permissions.kt (пример для сервиса доставки)Kotlin
// Пример из сервиса доставки — не твой код
data class DeliveryPermissions(
    val canPlaceOrder: Boolean = false,
    val canViewAllOrders: Boolean = false,
    val canManageRestaurants: Boolean = false,
    val canDeliverOrders: Boolean = false,
) {
    companion object {
        val GUEST = DeliveryPermissions()
        
        val CUSTOMER = DeliveryPermissions(
            canPlaceOrder = true
        )
        val COURIER = DeliveryPermissions(
            canDeliverOrders = true,
            canViewAllOrders = true
        )
        val RESTAURANT_ADMIN = DeliveryPermissions(
            canPlaceOrder = true,
            canViewAllOrders = true,
            canManageRestaurants = true
        )
    }
}
 
// Маппинг роль → права:
fun DeliveryRole.toPermissions(): DeliveryPermissions = when (this) {
    DeliveryRole.Customer        -> DeliveryPermissions.CUSTOMER
    DeliveryRole.Courier         -> DeliveryPermissions.COURIER
    DeliveryRole.RestaurantAdmin -> DeliveryPermissions.RESTAURANT_ADMIN
}

Фильтр авторизации: от userId к Permissions

▼

После фильтра аутентификации в Request лежит userId (или null). Второй фильтр — авторизации — берёт userId, находит пользователя в базе, получает его роль, конвертирует в права и кладёт в Request.

NotePermissionsFilter.kt (пример)Kotlin
fun makePermissionsFilter(
    storage: NoteStorage,
    userIdKey: /* RequestKey */,
    permissionsKey: /* RequestKey */
): Filter = Filter { next ->
    { request ->
        // Достаём userId из контекста (положил предыдущий фильтр)
        val userId: String? = userIdKey(request)
 
        // Получаем права: если нет userId — гость, иначе ищем пользователя
        val permissions = if (userId == null) {
            NotePermissions.GUEST
        } else {
            val user = storage.getUser(UUID.fromString(userId))
            user?.role?.toPermissions() ?: NotePermissions.GUEST
        }
 
        // Кладём права в Request и передаём дальше
        next(request.with(permissionsKey of permissions))
    }
}
ℹ️
Если userId есть, но пользователь не найден в базе — токен валидный, но пользователь удалён. Корректно вернуть GUEST или 401. В лабе, скорее всего, достаточно вернуть GUEST.

Маршрутизация + Filters

Авторизация маршрутов: 401 и 403

🚦

Теория: 401 vs 403 — когда и что возвращать

▼

401 Unauthorized

Мы не знаем кто делает запрос. Токена нет, или он истёк/повреждён.

Ответ: «Представьтесь, пожалуйста»

403 Forbidden

Мы знаем кто делает запрос, но у него нет прав на это действие.

Ответ: «Вам сюда нельзя»

Пример

Ты пришёл в банк без паспорта → 401 (не знаем кто ты). Ты пришёл с паспортом, но хочешь снять деньги с чужого счёта → 403 (знаем кто ты, но это чужое).

🔧

Паттерн: фильтр проверки прав для маршрута

▼

Создаём переиспользуемую функцию-фабрику, которая принимает условие и возвращает фильтр:

AuthorizationFilters.kt (пример)Kotlin
// Универсальный фильтр проверки права доступа к маршруту
fun requirePermission(
    userIdKey: /* RequestKey */,
    permissionsKey: /* RequestKey */,
    canAccess: (Permissions) -> Boolean,
): Filter = Filter { next ->
    { request ->
        val userId = userIdKey(request)
        val permissions = permissionsKey(request)
 
        when {
            // Токена нет — 401
            userId == null ->
                Response(Status.UNAUTHORIZED)
                    .header("Content-Type", "application/json")
                    .body("""{"Error": "Необходима авторизация"}""")
 
            // Токен есть, но прав нет — 403
            !canAccess(permissions) ->
                Response(Status.FORBIDDEN)
                    .header("Content-Type", "application/json")
                    .body("""{"Error": "Доступ запрещён"}""")
 
            // Всё хорошо — передаём дальше
            else -> next(request)
        }
    }
}

Применение в маршрутизаторе:

NoteRoutes.kt (пример применения)Kotlin
fun noteRoutes(storage: NoteStorage, ...): RoutingHttpHandler = routes(
 
    // Публичный маршрут — никаких фильтров авторизации
    "/notes" bind Method.GET to ListNotesHandler(storage),
 
    // Нужна аутентификация (любой авторизованный)
    "/notes" bind Method.POST to
        requirePermission(userIdKey, permissionsKey) { it.isAuthenticated }
            .then(CreateNoteHandler(storage, userIdKey)),
 
    // Нужна конкретная роль (только администратор)
    "/admin/users/{id}" bind Method.DELETE to
        requirePermission(userIdKey, permissionsKey) { it.canManageUsers }
            .then(DeleteUserHandler(storage, userIdKey)),
)
ℹ️
.then() в http4k — это цепочка фильтров. filter.then(handler) = «сначала прогони через filter, потом через handler».

📝 Что делаешь в своём проекте на этом этапе

  • Создай auth/AuthorizationFilters.kt с функцией requirePermission() (или несколькими специализированными функциями)
  • Создай новые v3-обработчики или скопируй v2 — в web/routes/v3/
  • В ApplicationRoutes.kt добавь маршруты /v3/... с соответствующими фильтрами авторизации перед каждым защищённым маршрутом
  • Убедись что маршруты без требований авторизации (GET /v3/tasks, GET /ping) работают без токена
  • Подключи authFilter.then(permissionsFilter).then(router) в сборке приложения в Main.kt

Проверка владения ресурсом (Owner Check)

👤

Теория: почему Owner — особый случай

▼

Проверка роли в фильтре работает без знания конкретного ресурса. Но для проверки владения нужно знать: «является ли текущий пользователь автором этой конкретной задачи?»

Это невозможно сделать в общем фильтре — нужно загружать задачу из базы. Поэтому такая проверка выполняется внутри обработчика.

Уровни авторизации
Фильтр
«есть токен?»
→
Фильтр маршрута
«есть права?»
→
Обработчик
«владеет ресурсом?»
UpdateNoteHandler.kt (пример — сервис заметок)Kotlin
class UpdateNoteHandler(
    private val storage: NoteStorage,
    private val userIdKey: /* ... */
) : HttpHandler {
 
    override fun invoke(request: Request): Response {
        // Достаём ID заметки из пути /notes/{note-id}
        val noteId = notIdLens(request)
 
        // Достаём текущего пользователя из контекста
        val currentUserId = UUID.fromString(userIdKey(request)!!)
 
        // Находим заметку
        val note = storage.getNote(noteId)
            ?: return Response(Status.NOT_FOUND)
                .body("""{"Error": "Заметка не найдена"}""")
 
        // ПРОВЕРКА ВЛАДЕНИЯ — является ли пользователь автором?
        if (note.authorId != currentUserId) {
            return Response(Status.FORBIDDEN)
                .body("""{"Error": "Нельзя редактировать чужую заметку"}""")
        }
 
        // Только теперь — обновляем заметку
        // ... парсинг тела запроса, сохранение
    }
}

Паттерн: сначала найти ресурс (404 если нет), затем проверить владение (403 если не владелец), затем обрабатывать.

val category = storage.getCategory(categoryId)
    ?: return Response(Status.NOT_FOUND)
        .body("""{"Error": "Категория не найдена"}""")
 
val permissions = permissionsKey(request)
val currentUserId = UUID.fromString(userIdKey(request)!!)
 
// Проверка: (текущий пользователь — владелец) ИЛИ (есть роль менеджера)
val isOwner = category.owner == currentUserId
val isManager = permissions.canManageCategories
 
if (!isOwner && !isManager) {
    return Response(Status.FORBIDDEN)
        .body("""{"Error": "Нет доступа"}""")
}
 
// Удаляем сначала все задачи категории, потом саму категорию
storage.getTasksByCategory(categoryId).forEach { storage.removeTask(it.id) }
storage.removeCategory(categoryId)
return Response(Status.NO_CONTENT)

📝 Что делаешь в своём проекте на этом этапе

  • В обработчике UpdateTaskHandler — добавь проверку: task.author != currentUserId → 403
  • В обработчике DeleteCategoryHandler — добавь проверку владения или роли: !isOwner && !isManager → 403
  • В обработчике UpdateCategoryHandler — аналогично
  • В обработчиках EisenhowerTasksHandler, TasksByTimeHandler, TasksStatisticsHandler — фильтруй задачи только по текущему пользователю (filter { it.author == currentUserId })
  • При создании задачи CreateTaskHandler — автор задачи = currentUserId из контекста (не из тела запроса!). Если категория передана — проверь что она принадлежит этому пользователю, иначе 403

Как собрать всё вместе в Main.kt

▼

Все части создаются один раз и передаются через dependency injection (просто параметрами в конструктор). Вот как это выглядит в NoteApp:

main() — сборка приложения (пример)Kotlin
fun main(args: Array<String>) {
    val appArgs = NoteArgs()
    JCommander.newBuilder().addObject(appArgs).build().parse(*args)
 
    val storage = NoteStorage(...)
 
    // 1. Создаём инструменты JWT
    val jwt = NoteJwtTools(appArgs.secret)
 
    // 2. Создаём ключи для контекста
    val userIdKey = RequestKey.optional<String>("userId")
    val permissionsKey = RequestKey.required<Permissions>("permissions")
 
    // 3. Создаём фильтры
    val authFilter = makeAuthFilter(jwt, userIdKey)
    val permissionsFilter = makePermissionsFilter(storage, userIdKey, permissionsKey)
 
    // 4. Создаём маршрутизатор
    val router = noteRoutes(storage, userIdKey, permissionsKey)
 
    // 5. Собираем цепочку фильтров
    val app = lensFailureFilter
        .then(authFilter)         // сначала читаем токен
        .then(permissionsFilter)  // потом считаем права
        .then(router)             // потом маршрутизируем
 
    app.asServer(Netty(appArgs.port)).start().block()
}
